## 架构演进记录

### v0.1 → v0.2：引入实体理解层

#### 问题发现过程

v0.1 在测试"深圳出差住宿费"时，分类步骤命中了"出差费用明细表"而非城市分类规则，导致第二步查询词变成垃圾内容。

**第一次修复尝试（错误）**：在 Planner prompt 中写入业务规则——"城市查询先用 search_classification 再用 search_standards"。
- 问题：把业务逻辑写进了通用 Planner，违反领域无关原则。

**第二次修复尝试（错误）**：在复合 Skill 内硬编码固定分类查询词——`"城市类别 A类城市 B类城市 C类城市"`。
- 问题：文档若写"一类/二类"或"甲级/乙级"立刻失效，换汤不换药。

**第三次修复尝试（错误）**：改为 `query + "地区分类 A类 B类 C类"`。
- 问题：同上，仍然假设了分类体系的命名方式。

#### 根因分析

上述三次都是"解题思路"——针对当前报销场景的局部修补。真正的问题是：

```
向量检索做的是语义相似，无法区分：
  "深圳" 是一个需要去找分类规则的【地区实体】
  还是一个普通的查询关键词

BM25 能找到包含"深圳"的 chunk，但：
  "揭阳"不在任何显式列表（只在"C类：其他"里）
  BM25 同样无效
```

这不是检索策略的问题，是**系统缺少对实体类型的理解**。

#### 新方案：实体理解层

| 维度 | v0.1 旧方案 | v0.2 新方案 |
|------|------------|------------|
| **出发点** | 如何找到正确的 chunk | 先理解查询里有什么实体、是什么类型 |
| **分类查询** | 硬编码查询词（A类/B类/C类）| 由实体类型决定查什么规则，与文档命名无关 |
| **揭阳这类隐式实体** | 无法处理 | 找到"C类：其他"兜底规则后，交由 LLM 推断 |
| **跨业务复用** | 每个业务场景单独修补 | 实体理解是通用能力，适用所有分类查询场景 |
| **文档变化适应性** | 文档改命名方式即失效 | 只关心实体类型，不关心文档如何命名分类 |

#### 通用性说明

同样的结构性问题存在于所有涉及分类体系的业务场景：

| 用户问题 | 实体 | 实体类型 | 需要找的规则类型 |
|---------|------|---------|----------------|
| 深圳/揭阳出差住宿费 | 深圳/揭阳 | 地区名 | 地区分类规定 |
| P6员工年假天数 | P6 | 职级名 | 职级分类规定 |
| 研发部门加班补贴 | 研发部 | 部门名 | 部门分类规定 |
| 50万合同谁来审批 | 50万 | 金额数值 | 审批权限阈值 |
| 试用期员工能报销吗 | 试用期 | 员工状态 | 员工类型适用规定 |

**核心原则**：实体本身不是答案，实体所属类别对应的规则才是答案。实体理解层的职责是识别实体类型，而非关心文档用什么命名体系。

## 一、问题背景

### 当前方案的核心缺陷

`demo/` 中的 ReAct 方案（`retrieve_rag_with_reflection`）对**多层依赖问题**失效：

```
用户提问："深圳出差住宿费是多少？"

正确推理链：
  深圳 → 查城市分类 → A类城市 → 查A类住宿标准 → 500元/天

当前失效原因：
  子问题并行生成，互相独立，第2步无法利用第1步的结果
  导致向量检索用"深圳住宿费"作为 query，而知识库里只有"A类城市住宿费"
```

### 根本诉求

让 LLM 像一个"会思考的检索员"，自己决定：
1. **要查什么**（理解问题的信息需求）
2. **按什么顺序查**（处理依赖关系）
3. **用什么方式查**（选择合适的检索策略）

## 二、架构设计

### 2.1 整体流程

#### v1.0 架构图

```
用户问题
    │
    ▼
┌─────────────────────────────┐
│   Step 1: 意图理解 + 计划制定  │  LLM 分析问题，输出结构化调用计划
│   (Planner)                 │  输出格式：JSON（skill 列表 + 参数）
└─────────────────────────────┘
    │
    ▼
┌─────────────────────────────┐
│   Step 2: Skill 串行执行      │  按顺序执行每个 skill
│   (Executor)                │  每步结果可作为下一步的输入参数
└─────────────────────────────┘
    │
    ▼
┌─────────────────────────────┐
│   Step 3: 信息充分性判断      │  LLM 判断已收集信息是否足够回答
│   (Evaluator)               │  不足则补充调用（最多3轮）
└─────────────────────────────┘
    │
    ▼
┌─────────────────────────────┐
│   Step 4: 最终回答生成        │  基于所有检索结果生成最终答案
│   (Generator)               │
└─────────────────────────────┘
```

> v1.0 局限：Planner 直接接收原始问题，不理解"深圳"是地区实体，导致检索策略无法自适应不同实体类型。业务规则只能硬编码在 Planner prompt 或 Skill 内部，换一套文档命名体系即失效。

#### v2.0 架构图（引入实体理解层）

```
用户问题
    │
    ▼

│                             │  例："深圳"→地区名，"住宿费"→费用类型
└─────────────────────────────┘
    │  携带实体类型信息
    ▼
┌─────────────────────────────┐
│   Step 1: 意图理解 + 计划制定  │  结合实体类型制定检索计划
│   (Planner)                 │  知道"深圳是地区名"→应查地区分类规则
└─────────────────────────────┘
    │
    ▼
┌─────────────────────────────┐
│   Step 2: Skill 串行执行      │  按顺序执行每个 Skill
│   (Executor)                │  每步结果可作为下一步的输入参数
└─────────────────────────────┘
    │
    ▼
┌─────────────────────────────┐
│   Step 3: 信息充分性判断      │  LLM 判断已收集信息是否足够回答
│   (Evaluator)               │  不足则补充调用（最多3轮）
└─────────────────────────────┘

    ▼
┌─────────────────────────────┐
│   Step 4: 最终回答生成        │  结合实体类型推断隐式分类

└─────────────────────────────┘
```

> v2.0 改进：实体理解层将"深圳/揭阳是地区名"在运行时动态识别，Planner 和 Skill 无需硬编码文档的分类命名方式（A/B/C 或 一/二/三），跨业务场景通用。

### 2.2 Skill 定义

Skill 分为两种类型：**原子 Skill** 和 **复合 Skill**。

#### 原子 Skill（领域无关的检索原语）

按**检索意图**划分，不包含业务逻辑，具有通用性：

| Skill 名称 | 描述 | 适用场景 | 推荐检索方式 |
|-----------|------|---------|------------|
| `search_classification` | 查询某实体属于哪个类别/等级/分组 | "深圳属于哪类城市"、"P6 对应什么职级" | BM25（精确匹配分类表格）|
| `search_standards` | 查询某类别下的具体数值/标准/限额 | "A类城市住宿费多少"、"专业版价格" | 向量检索（语义匹配费用相关内容）|
| `search_procedure` | 查询申请/审批/操作流程 | "怎么申请年假"、"报销流程" | 混合检索 |
| `search_policy` | 查询规章制度/适用条件/例外情形 | "出差有什么规定"、"哪些不能报销" | 混合检索 |
| `search_factual` | 直接事实检索（无依赖关系的简单问题）| "产品支持哪些浏览器" | 向量检索 |

#### 复合 Skill（领域级封装）

将特定业务场景的多步检索链封装在 Skill 内部，Planner 只需路由，不含业务规则：

| Skill 名称 | 描述 | 内部步骤 |
|-----------|------|---------|
| `search_expense_reimbursement` | 查询出差住宿费、餐饮补贴等财务报销标准 | ①查城市/职级分类 → ②查对应费用标准，结果合并返回 |

> **设计原则**：业务领域知识封装在复合 Skill 内部，Planner prompt 保持领域无关。
> 新增业务场景只需添加新的复合 Skill 文件，无需修改 Planner。

### 2.3 LLM 调用计划格式（Planner 输出）

```json
{
  "reasoning": "用户问深圳出差住宿费，需要先确认深圳的城市分类，再根据分类查对应标准",
  "steps": [
    {
      "step_id": 1,
      "skill": "search_classification",
      "query": "深圳 城市分类 出差",
      "depends_on": null
    },
    {
      "step_id": 2,
      "skill": "search_standards",
      "query": "<step_1_result> 住宿费标准",
      "depends_on": 1
    }
  ]
}
```

**关键机制：`depends_on` + `<step_N_result>` 占位符**
- `depends_on: 1` 表示执行前必须等 step_1 完成
- `<step_N_result>` 在执行时被替换为对应步骤检索结果的关键实体摘要
- Executor 遍历所有已完成步骤进行替换，支持 query 中同时引用多个前置步骤
- 摘要提取会清洗 Markdown 表格格式符号（去除 `|`、`---` 等），避免污染后续查询词

## 四、核心模块设计

### 4.1 `skills.py` — Skill 注册与执行

```python
# 伪代码示意
SKILL_REGISTRY = {
    "search_classification": {
        "description": "查询某个实体（城市/职级/类型）属于哪个分类/等级",
        "retrieval_method": "bm25",           # 关键词优先
    },
    "search_standards": {
        "description": "查询某个分类下的具体标准/金额/限额",
        "retrieval_method": "vector",          # 语义优先
    },
    "search_procedure": {
        "description": "查询某个业务的申请/审批流程步骤",
        "retrieval_method": "hybrid",
    },
    "search_policy": {
        "description": "查询规章制度/政策条款/适用条件",
        "retrieval_method": "hybrid",
    },
    "search_factual": {
        "description": "直接查询事实性信息",
        "retrieval_method": "vector",
    },
}
```

### 4.2 `agentic_rag.py` — 主流程

```
阶段1: Planner
  - 输入：用户问题 + SKILL_REGISTRY 描述
  - 输出：结构化 JSON 调用计划

阶段2: Executor
  - 遍历 steps，处理 depends_on 和占位符替换
  - 调用对应 Skill 的检索函数
  - 收集每步检索结果

阶段3: Evaluator（可选，轮数控制）
  - 判断信息是否充分
  - 不充分则追加新的 skill 调用

阶段4: Generator
  - 将所有检索结果 + 原始问题喂给 LLM
  - 生成最终回答
```

## 五、关键设计决策与待讨论点

### ✅ 已确定

1. **Skill 按检索意图划分**，不绑定业务领域，新增文档无需修改 Skill 定义
2. **串行执行**，支持 `depends_on` 依赖链，解决多层推理问题
3. **最多3轮** Planner → Executor 循环，防止死循环
4. **LLM 统一用 Ollama**（与 demo 保持一致），后续可扩展 OpenAI

### ❓ 待讨论

1. **BM25 实现方式**：用 `rank_bm25` 库还是 Elasticsearch？
   - 用 `rank_bm25`（纯 Python，无需额外服务）
2. **`{{step_N_result}}` 的摘要方式**：检索结果有多个 chunk，如何提炼出"A类城市"这样的关键词？
   - 方案A：让 LLM 对检索结果做一次提炼再传入下一步
3. **向量库**：复用 ChromaDB 还是换 FAISS？
   - 复用 ChromaDB，避免重复造轮子
4. **Streamlit 前端**：是否在这个阶段加？
   - streamlit前端，加入不复杂，可以直接加

## 八、编码规范

> 所有代码编写均需遵守本节规范，优先级高于实现速度。

### 8.1 设计原则

**高内聚低耦合**
- 每个模块只做一件事：`agentic_rag.py` 负责流程编排，`skills/` 负责检索，`app.py` 只负责 UI 展示
- 模块间通过明确的接口通信，不直接访问彼此内部状态
- Skill 通过依赖注入（`retriever` 参数）获取检索能力，不直接 import `agentic_rag`

**可读性优先**
- 函数职责单一，超过 40 行考虑拆分
- 变量和函数命名清晰表达意图，避免缩写（`exec` → `execute_plan`）
- 关键逻辑处写注释，说明"为什么"而非"是什么"

### 8.2 设计模式参考

| 模式 | 应用位置 | 说明 |
|------|---------|------|
| **策略模式（Strategy）** | `Retriever` 的检索方法 | `vector_search` / `bm25_search` / `hybrid_search` 可互换，Skill 通过 `retrieval_method` 字段选择 |
| **工厂模式（Factory）** | `skills/__init__.py` 动态加载 | 根据文件名自动注册 Skill，调用方无需感知具体实现 |
| **职责链（Chain of Responsibility）** | Planner → Executor → Generator | 每个阶段只处理自己的职责，结果传递给下一阶段 |
| **依赖注入（DI）** | `execute(query, retriever)` | Skill 不持有 Retriever 实例，由 Executor 在调用时注入 |

### 8.3 TDD 指引

对以下核心逻辑，**先写测试再写实现**：

| 测试文件 | 覆盖的函数 | 典型测试用例 |
|---------|-----------|------------|
| `tests/test_chunking.py` | `_chunk_markdown` / `_extract_blocks` | 表格不被截断、表格内空行不中断 |
| `tests/test_retriever.py` | `Retriever.vector_search` | 返回格式正确（含 text/source/score）、空查询不崩溃 |
| `tests/test_executor.py` | `execute_plan` | `depends_on` 顺序正确、占位符替换正确、Skill 不存在时不崩溃 |
| `tests/test_planner.py` | `plan` | LLM 返回非法 JSON 时返回 error 字段、steps 为空时正常处理 |

测试运行命令：`pytest tests/ -v`

> 注：LLM 调用在测试中使用 Mock，不依赖真实 Ollama 服务。
