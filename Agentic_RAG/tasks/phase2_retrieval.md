# Phase 2 — 检索增强

> 目标：为不同 Skill 配备最优检索策略，提升召回质量
> 前置条件：Phase 1 全部任务通过验收
> 验收方式：对比 Phase 1 的检索结果，人工判断质量是否提升

---

## 架构演进背景

### v1.0 的局限

Phase 1 全部使用向量检索。向量检索做语义相似匹配，对以下场景效果差：
- **精确实体查找**："深圳" 在向量空间与"费用申请表"距离更近，而非城市分类表
- **关键词敏感内容**：分类表格、人名、金额阈值等，需要精确匹配而非语义近似

### BM25 的价值与边界

BM25 关键词检索解决的是**精确匹配**问题：
- ✅ "深圳" → 命中"A类地区：北京、上海、广州、**深圳**"
- ❌ "揭阳" → 揭阳不在任何显式列表，BM25 同样找不到

> **重要**：BM25 是 Phase 2 的核心，但不是终点。揭阳这类隐式实体的处理，依赖 Phase 3 实体理解层。Phase 2 的目标是打好检索基础设施，Phase 3 在此之上做实体语义理解。

## T2-1 实现 BM25 检索

**任务：** 在 `Retriever` 中增加 `bm25_search` 方法，使用 `rank_bm25` 库

**完成标准：**
- [ ] `pip install rank-bm25` 后功能可用
- [ ] `retriever.bm25_search("深圳", top_k=5)` 返回格式与 `vector_search` 相同（含 text/source/score 字段）
- [ ] 人工验证：搜索"深圳"时，BM25 Top-1 结果包含"深圳"关键词（命中城市分类 chunk）
- [ ] 人工验证：搜索"揭阳"时，BM25 无法命中（返回空或低相关结果），记录此边界，留给 Phase 3 处理

---

## T2-2 实现混合检索（Hybrid）

**任务：** 在 `Retriever` 中增加 `hybrid_search` 方法，用 RRF（倒数排名融合）合并 BM25 和向量检索结果

**RRF 公式：** `score = Σ 1 / (k + rank_i)`，k=60

**完成标准：**
- [ ] `retriever.hybrid_search("深圳住宿费", top_k=5)` 返回格式正确
- [ ] 人工验证：混合结果中同时出现分类规则和费用标准相关的 chunk（两路各有贡献）
- [ ] 单路检索返回空时，混合检索不崩溃，降级为另一路结果

---

## T2-3 为各 Skill 绑定最优检索策略

**任务：** 根据各 Skill 的检索意图，在 `SKILL_META` 中设置 `retrieval_method`，并在 `execute` 中调用对应方法

| Skill | 推荐策略 | 理由 |
|-------|---------|------|
| `search_classification` | `bm25` | 分类表格含明确关键词，精确匹配效果好；对隐式实体（揭阳）的处理留给 Phase 3 |
| `search_standards` | `vector` | 费用描述语义多样，向量检索覆盖更广 |
| `search_procedure` | `hybrid` | 流程类内容兼具关键词和语义需求 |
| `search_policy` | `hybrid` | 政策条款同上 |
| `search_factual` | `vector` | 事实类问题语义检索为主 |
| `search_expense_reimbursement` | `composite`（内部自定义）| 复合 Skill，分类步骤用 BM25，标准步骤用向量 |

**完成标准：**
- [ ] 各 Skill 的 `SKILL_META["retrieval_method"]` 与上表一致
- [ ] 日志中可见各步骤使用的检索方式标识
- [ ] 人工验证：搜索"深圳出差住宿费"，分类步骤用 BM25 命中城市分类 chunk
