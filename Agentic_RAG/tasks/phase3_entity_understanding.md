# Phase 3 — 业务语义补全与结构化拆解

> 原名：实体理解层。经过实际测试与深入讨论，扩展为完整的查询语义解析能力。
> 目标：让系统在检索之前，真正"读懂"用户问题——补全缺失维度、识别约束条件、判断意图颗粒度、检测歧义与冲突
> 前置条件：Phase 2 全部任务通过验收
> 验收方式：多类型问题均能正确处理，包括隐式主语推断、约束条件过滤、歧义主动追问、冲突预警

---

## 架构背景

### 问题的本质：用户问题天然不完整

企业查询普遍存在**语义缺失**和**隐含约束**：

```
"深圳出差住宿的标准"
  ✅ Where = 深圳
  ✅ What  = 住宿费标准
  ❌ Who   = 缺失（职级/身份）
  → 副总裁：800元/天 vs 其他员工：420元/天，答案完全不同

"2023年后入职员工在非工作日出差的补贴"
  ✅ Who   = 员工
  ✅ What  = 出差补贴
  ❌ 时间约束 = 2023年后入职（不是查2023年的政策！）
  ❌ 场景约束 = 非工作日
  → 不带约束检索，可能返回旧政策或工作日标准
```

### 五类语义理解能力

| 能力 | 描述 | 工程意义 |
|------|------|---------|
| **维度完整性** | Who/Where/What 三维识别与缺失检测 | 决定检索策略和默认假设 |
| **约束条件识别** | 时间约束、场景约束、条件约束 | 约束词必须带入检索 Query，防止召回错误政策 |
| **意图颗粒度** | 总结性回答 vs 精确事实查找 | 影响 Generator 的组织策略 |
| **歧义与不完整** | 关键要素缺失时，先问用户再检索 | 避免盲目检索后给出错误答案 |
| **冲突检测** | 识别问题中逻辑相悖的实体组合 | Planner 预警 or Generator 提醒 |

## T3-1 实现 `query_understanding.py` — 结构化语义解析

**任务：** 新建模块，对用户查询执行完整语义解析，输出结构化 `QueryStructure`

**输出格式：**

```json
{
  "original": "深圳出差住宿的标准",
  "expanded": "其他员工（默认）到深圳出差，住宿标准金额是多少",
  "dimensions": {
    "who":   {"value": "其他员工", "inferred": true},
    "where": {"value": "深圳",    "inferred": false},
    "what":  {"value": "住宿费标准","inferred": false}
  },
  "entities": [
    {"text": "深圳",   "type": "地区名", "role": "查询对象"},
    {"text": "住宿费", "type": "费用项", "role": "目标属性"}
  ],
  "constraints": [
    {"type": "时间约束", "value": "2023年后入职", "source": "显式"},
    {"type": "场景约束", "value": "非工作日",     "source": "显式"}
  ],
  "intent_granularity": "精确事实",
  "missing": ["who"],
  "conflicts": [],
  "needs_clarification": false
}
```

**实现方式：** 调用 LLM，结构化 prompt 提取，输出 JSON

**完成标准：**
- [ ] `parse_query("深圳出差住宿的标准")` → Who 缺失，entities 含深圳(地区名/查询对象)
- [ ] `parse_query("揭阳出差住宿的标准")` → 与深圳同结构，Where=揭阳
- [ ] `parse_query("2023年后入职员工非工作日出差补贴")` → constraints 含时间约束+场景约束
- [ ] `parse_query("简单说说出差流程")` → intent_granularity="总结性"
- [ ] `parse_query("报销单第二联贴在哪")` → intent_granularity="精确事实"
- [ ] `parse_query("住宿费标准是多少")` → missing=["who","where"]，needs_clarification=true
- [ ] `parse_query("P10职级实习生出差补贴")` → conflicts 非空，检测到职级与身份冲突
- [ ] LLM 返回非法 JSON 时降级返回原始 question，不崩溃
- [ ] 有对应单元测试（Mock LLM）

## T3-2 歧义与不完整处理 — 先问再查

**任务：** `needs_clarification=true` 时，系统直接向用户追问缺失信息，而非盲目检索

**完成标准：**
- [ ] 当 missing 包含影响答案的关键维度（如 who 或 where）时，`needs_clarification=true`
- [ ] 系统返回追问而非检索结果，例如："请问您要查询哪个城市的标准？" 或 "请问您的职级是？"
- [ ] 追问文案从 `missing` 字段自动生成，不硬编码
- [ ] 用户补充信息后，系统用完整问题重新走全流程

---

## T3-3 约束条件注入检索

**任务：** 将 `constraints` 中的约束词显式带入 Planner 生成的检索 Query

**完成标准：**
- [ ] 对含时间约束的问题，检索 Query 中携带时间限定词，防止召回旧版政策
- [ ] 对含场景约束的问题（如"非工作日"），检索 Query 包含该约束
- [ ] Planner prompt 中明确说明：constraints 字段的内容必须反映在每一个检索步骤的 query 中

---

## T3-4 意图颗粒度影响 Generator 策略

**任务：** Generator 根据 `intent_granularity` 调整回答组织方式

**完成标准：**
- [ ] `总结性` → Generator prompt 指示：合并多个 chunk 要点，给出结构化摘要
- [ ] `精确事实` → Generator prompt 指示：精确定位具体数字或规则，不展开背景
- [ ] 人工对比两类问题的回答风格，确认差异明显

---

## T3-5 冲突检测与预警

**任务：** 识别问题中逻辑相悖的实体组合，在 Planner 或 Generator 阶段预警

**完成标准：**
- [ ] `conflicts` 非空时，Generator 在回答末尾附加提醒（如"检测到 P10 与实习生身份存在冲突，请核实"）
- [ ] 冲突不导致系统拒绝回答，仍尽力检索并提供相关信息

---

## T3-6 实体类型驱动 Skill 选择

**任务：** 更新各 Skill 的 description，改为以**实体类型**描述适用场景，而非举具体业务例子

**背景：** Planner 看到的应该是"地区名"，而非"深圳"；是"职级名"，而非"P6"。
这样 Planner 能通用路由，不依赖 prompt 中列举具体词汇。

**更新示例：**

```python
# 旧（举例式，耦合具体业务词）
"description": "查询城市如深圳、广州属于哪类出差地区"

# 新（类型驱动，通用）
"description": "当 query 中包含【地区名】、【职级名】等需要确认所属规则类别的实体时调用，返回该实体的分类信息"
```

**完成标准：**
- [ ] 所有 Skill 的 description 改为实体类型描述
- [ ] 人工验证：Planner 对"揭阳出差住宿费"和"深圳出差住宿费"选择相同的 Skill 组合
- [ ] Skill description 中无"深圳"、"广州"、"A类"等具体业务词汇
