# Phase 4 — 健壮性

> （原 Phase 3，因引入实体理解层后整体阶段顺延）
> 目标：处理异常情况，确保系统在边界条件下不崩溃、不产生错误回答
> 前置条件：Phase 3 全部任务通过验收
> 验收方式：人工构造异常输入，观察系统行为

---

## T3-1 Planner 输出容错

**任务：** 处理 LLM 输出非法 JSON 或缺字段的情况

**需覆盖的异常场景：**
- LLM 输出的 JSON 格式不合法（缺括号、多余文字）
- `steps` 字段为空列表
- 某个 step 引用了不存在的 Skill 名称
- `depends_on` 指向不存在的 `step_id`

**完成标准：**
- [ ] 以上4种场景均不导致程序崩溃
- [ ] 每种场景都有清晰的错误日志，说明具体原因
- [ ] JSON 解析失败时，有1次重试机会（重新调用 LLM）；重试仍失败则回答"系统处理异常，请重新提问"

---

## T3-2 Evaluator（信息充分性判断）

**任务：** 在 Executor 执行完毕后，由 LLM 判断已收集信息是否足以回答问题；不足则触发补充检索

**完成标准：**
- [ ] Evaluator 输出格式为 `{"sufficient": true/false, "missing": "缺少什么信息"}`
- [ ] `sufficient=false` 时，Planner 根据 `missing` 字段补充生成新的 step 并执行
- [ ] 补充检索**最多触发2次**（总轮数上限3），超出后强制进入 Generator
- [ ] 日志中可见 `[Evaluator] 信息充分，进入生成阶段` 或 `[Evaluator] 信息不足：xxx，补充检索`

---

## T3-3 知识库边界处理

**任务：** 确保知识库中确实不存在的信息不被编造

**验收问题：**
- "火星出差补贴是多少？"（完全不存在的内容）
- "CEO 的联系方式是什么？"（知识库未收录）

**完成标准：**
- [ ] 以上问题的回答中，不包含任何捏造的具体数字或事实
- [ ] 回答明确说明"知识库中未找到相关信息"
- [ ] Evaluator 在检索结果相关性极低时，不误判为 `sufficient=true`
